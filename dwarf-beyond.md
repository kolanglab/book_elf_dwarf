# 扱わなかった DWARF ―― 仕様の残りの地図

前章「何が分かり、何が分からないか」では、DWARF をもってしても原理的に届かない**境界**を見ました。この章は趣を変え、DWARF 仕様そのもののうち、**本書があえて扱わなかった部分**を地図として見渡します。前者が「DWARF の限界」だとすれば、こちらは「本書の射程の限界」です。

DWARF 第 5 版の仕様書 [](#cite:dwarf2017) は付録まで含めて 400 ページ近くあり、本書が触れたのはその中核 ―― DIE の符号化、型と場所式、行番号情報 ―― にすぎません。ここでは残りの主要領域について、**何のためのものか・実務でどこに現れるか・調べるときの手がかり**を簡潔に示します。「この先に何があるか」を知っておけば、`readelf --debug-dump` の見慣れないセクション名に出会っても、見当をつけて仕様書の該当章へ飛べます。

## 高速化のための索引セクション

`.debug_info` を毎回先頭から走査するのは、大きなプログラムでは遅すぎます。そこで DWARF は、検索を速くするための**索引**を別セクションに用意します。

| セクション | 役割 |
|---|---|
| `.debug_aranges` | 「このアドレス範囲はどの CU に属するか」の表。アドレスから CU を即座に絞り込む |
| `.debug_names` | 名前（関数名・型名など）から DIE を引く、DWARF 第 5 版の正式な名前索引 |
| `.debug_pubnames` / `.debug_pubtypes` | 第 4 版までの旧式の名前索引（第 5 版で `.debug_names` に置き換え） |

デバッガが「`main` という関数を名前で探す」「このアドレスを含む CU はどれか」を高速に行えるのは、これらの索引のおかげです。索引が無くても情報は `.debug_info` から復元できますが、その場合は全 CU の走査が必要になります。

## 場所式の全体像 ―― スタックマシンの命令セット

第 10 章では場所式 (`DW_OP_*`) のうち、`DW_OP_addr`・`DW_OP_fbreg`・`DW_OP_reg` といった基本的なものだけを紹介しました。しかし DWARF 式は、変数の場所を指すだけの小さな記法ではありません。スタックを持つ**汎用の計算機**であり、四則演算から条件分岐、サブルーチン呼び出し、型変換までを備えた、小さなプログラミング言語です。本書の方針からすればここも概略で済ませたいところですが、これだけ多くのオペコードを持つと述べた以上、看板倒れにならないよう、この命令セットだけは全体を一覧しておきます。オペコードは、働きによって次のような系統に分けられます。

まず、値をスタックに積むためのオペコードです。小さな定数を 1 バイトで積むものから、`.debug_addr` 索引を経由するもの（Split DWARF 用）まであります。

| オペコード | 働き |
|---|---|
| `DW_OP_lit0`–`lit31` | 0〜31 の小さな定数を 1 バイトで積む |
| `DW_OP_addr` | 絶対アドレスを積む |
| `DW_OP_const1u`/`1s` 〜 `const8u`/`8s` | 1〜8 バイトの定数を積む |
| `DW_OP_constu` / `consts` | LEB128 の符号なし／符号付き定数を積む |
| `DW_OP_addrx` / `constx` | `.debug_addr` の索引経由でアドレス／定数を積む |

次に、レジスタやフレームを基準に値を積むものです。第 10 章で見た `DW_OP_fbreg` はこの系統の一つでした。

| オペコード | 働き |
|---|---|
| `DW_OP_fbreg` | フレームベース + オフセットを積む |
| `DW_OP_breg0`–`breg31` | 指定レジスタの値 + オフセットを積む |
| `DW_OP_bregx` | 任意番号のレジスタの値 + オフセットを積む |

スタックそのものを操作したり、積んだアドレスからメモリを読んだりするものもあります。

| オペコード | 働き |
|---|---|
| `DW_OP_dup`, `drop`, `over`, `pick`, `swap`, `rot` | スタックの複製・破棄・並べ替え |
| `DW_OP_deref`, `deref_size` | 頂上をアドレスとみなしメモリから値を読む |
| `DW_OP_xderef`, `xderef_size` | アドレス空間 ID を伴うメモリ参照 |
| `DW_OP_push_object_address` | いま評価中のオブジェクトのアドレスを積む |
| `DW_OP_form_tls_address` | スレッドローカル変数の実アドレスを解決する |
| `DW_OP_call_frame_cfa` | CFI が定める CFA（フレーム基準アドレス）を積む |

ここからが「計算機」たるゆえんです。算術・論理演算がひととおりそろっています。

| オペコード | 働き |
|---|---|
| `DW_OP_abs`, `neg`, `not` | 絶対値・符号反転・ビット反転 |
| `DW_OP_and`, `or`, `xor` | ビット論理積・論理和・排他的論理和 |
| `DW_OP_plus`, `minus`, `mul`, `div`, `mod` | 加減乗除と剰余 |
| `DW_OP_plus_uconst` | 頂上に定数を加える（頻出する加算の短縮形） |
| `DW_OP_shl`, `shr`, `shra` | 左／右シフト、算術右シフト |

比較と制御フロー ―― 条件分岐や、別の DIE の式をサブルーチンとして呼ぶ命令まであります。

| オペコード | 働き |
|---|---|
| `DW_OP_eq`, `ne`, `lt`, `le`, `gt`, `ge` | 2 値を比較し、真偽（1/0）を積む |
| `DW_OP_skip` | 式中の別位置へ無条件に飛ぶ |
| `DW_OP_bra` | 頂上が非 0 なら飛ぶ条件分岐 |
| `DW_OP_call2`, `call4`, `call_ref` | 別の DIE が持つ式をサブルーチンとして呼ぶ |

第 5 版では、値に型情報を伴わせる「型付き演算」が加わりました。浮動小数の途中計算などを正しく扱うためのものです。

| オペコード | 働き |
|---|---|
| `DW_OP_const_type` | 型情報付きの定数を積む |
| `DW_OP_regval_type` | 型情報付きでレジスタ値を積む |
| `DW_OP_deref_type`, `xderef_type` | 型情報付きでメモリを読む |
| `DW_OP_convert`, `reinterpret` | 頂上の値を別の型へ変換／再解釈する |

最後に、「値そのもの」ではなく「**場所**」を表すための特別なオペコード群です。最適化下のデバッグで鍵になるのはここです。

| オペコード | 働き |
|---|---|
| `DW_OP_reg0`–`reg31`, `regx` | 値はレジスタの中にあり、メモリ上のアドレスを持たない |
| `DW_OP_implicit_value` | 値そのものを式の中に直接埋め込む |
| `DW_OP_stack_value` | 計算結果そのものが値（どこにも格納されていない） |
| `DW_OP_implicit_pointer` | 最適化で消えたポインタが指していた先を表す |
| `DW_OP_piece`, `bit_piece` | 値を複数の場所に分けて格納する（合成ロケーション） |
| `DW_OP_entry_value` | 関数に入った時点での、ある式の値 |
| `DW_OP_nop` / `DW_OP_lo_user`〜`hi_user` | 何もしない命令と、ベンダ拡張用の予約番号 |

これだけのオペコードがある、と聞くと過剰に思えるかもしれません。しかし、最適化されたコードで変数の在りかを正直に書こうとすると、この表現力が要るのです。実際に効いてくる代表例を挙げます。`DW_OP_stack_value` や `DW_OP_implicit_value` は、「値は計算で求まるが、メモリにもレジスタにも実体がない」状態を表します。たとえば `x * 2` という部分式が最適化で消え、`x` から導けるだけになったとき、デバッガはこの式を評価して値を**復元**します。`DW_OP_piece` は、構造体の前半はレジスタ・後半はスタック、というように 1 つの値が散らばった状況を表します。`DW_OP_entry_value` は、引数が関数の途中で上書きされても「入口での値」を呼び出し元から復元できるようにし、`DW_OP_implicit_pointer` は、ポインタ変数自体が消えても、それが指していた対象を追えるようにします。

> [!NOTE]
> DWARF 式は、場所の計算だけでなく、配列の動的な要素数や、データメンバの動的なオフセットなど、「コンパイル時には定数にならない値」の記述にも同じ言語が使われます。Fortran の可変長配列や C++ の一部の表現がその例です。第 10 章で見た `DW_OP_fbreg`・`DW_OP_reg` は、この奥行きのある言語の、ごく入口にすぎなかったわけです。

## アドレス範囲のリスト

関数や変数が、連続しない複数のアドレス範囲に散らばることがあります。最適化でコードが分割されたり、`DW_TAG_lexical_block`（局所スコープ）が飛び地になったりする場合です。この「範囲の集合」を表すのが**範囲リスト**で、`.debug_rnglists`（第 5 版。旧称 `.debug_ranges`）に格納され、`DW_AT_ranges` 属性から参照されます。`DW_AT_low_pc`/`DW_AT_high_pc` という単一の連続範囲では表せない、飛び地のアドレス範囲を記述するためのものだと考えてください。

## コールフレーム情報の中身

前章では CFI（`.debug_frame` / `.eh_frame`）を「スタックを巻き戻すための表を生成するバイトコード」として概観しました。バックトレースがフレームポインタなしでも正確にたどれる理由はそこで述べたとおりですが、その表が**どんな部品で組み立てられているか**には踏み込んでいません。ここで骨格だけ補っておきましょう。

CFI の本体は、**CIE** (Common Information Entry) と **FDE** (Frame Description Entry) という 2 種類のエントリから成ります。CIE は複数の関数に共通する設定 ―― アドレスの符号化方式や巻き戻しの既定ルールなど ―― を一度だけ定め、FDE は関数 1 つぶんの巻き戻し情報を、対応する CIE を参照しながら記述します。1 つの CIE を多数の FDE が共有する、という入れ子の関係です。

各 FDE の中身は、行番号プログラムと同じ発想 ―― **状態機械を駆動するバイトコード** ―― で書かれています。`DW_CFA_*` という命令の列が、アドレスを進めながら、「この区間では CFA（フレームの基準アドレス）はどのレジスタ + オフセットで求まるか」「戻りアドレスや退避されたレジスタは CFA からどこにあるか」を少しずつ更新していきます。第 11 章の行番号状態機械を理解していれば、その構造はほとんどそのまま頭に入るはずです。

なお `.eh_frame` は、この形式を例外処理向けに最適化した変種で、検索用の `.eh_frame_hdr` を伴います。`strip` 後のリリースバイナリにも残る点は前章で触れたとおりで、デバッグ情報なしでもバックトレースが出せる根拠になっています。自前でバックトレースを実装する、あるいは JIT が生成したコードのアンワインド情報を出力するといった場面では、この CIE/FDE と `DW_CFA_*` を正確に読み書きする必要が出てきます。

## 分割 DWARF と型の重複排除

デバッグ情報はしばしば本体のコードより大きくなります。これを管理するための仕組み群も、本書では概略に留めました。

| 仕組み | 関連セクション | 役割 |
|---|---|---|
| Split DWARF | `.debug_info.dwo` ほか `.dwo` セクション群 | デバッグ情報の大半を別ファイル (`.dwo`) へ追い出し、リンク時間と実行ファイルサイズを削減 |
| アドレス／文字列の間接化 | `.debug_addr`, `.debug_str_offsets` | 分割時にアドレスや文字列を一段の索引経由で参照する |
| パッケージファイル | `.debug_cu_index`, `.debug_tu_index`（`.dwp`） | 多数の `.dwo` を 1 ファイルにまとめる |
| 型単位 | `.debug_types`（第 4 版）/ `DW_UT_type`（第 5 版） | 同一の型定義を一度だけ記述し、各 CU から共有（COMDAT 型重複排除） |
| 補助オブジェクト | `.debug_sup`, DWZ | 複数バイナリ間で共通のデバッグ情報をくくり出す |

大規模な C++ プロジェクトをビルドすると、同じ `std::string` の型情報が何百もの翻訳単位に重複しがちです。**型単位**や DWZ は、この重複を畳んでデバッグ情報の肥大を抑えます。リンク時間やバイナリサイズに悩むときに効いてくる領域です。

> [!TIP]
> Split DWARF は「動かす ELF は小さく、理解する DWARF は別管理に」という、第 8 章でも触れた分離の流れの中心です。`-gsplit-dwarf` でビルドすると `.dwo` ファイルが生成され、最終バイナリには参照だけが残ります。CI やコンテナでデバッグ情報を別配布する構成で広く使われています。

## マクロと、版・形式のバリエーション

これまでの分類に収まらない、細々とした ―― しかし実務では確実に顔を出す ―― 三つの話題を最後にまとめておきます。マクロ情報、DWARF の版ごとの違い、そして内部オフセット幅の区別です。

- **マクロ情報** (`.debug_macro` / 旧 `.debug_macinfo`): `#define` の定義位置や展開を記録します。デバッガで「このマクロの定義は何か」を引くための情報ですが、生成するコンパイラや読むツールが限られ、実用は限定的です（前章でも触れたとおりです）。
- **DWARF のバージョン差**: 本書は第 5 版に統一しましたが、現役のバイナリには第 2〜4 版も多く残っています。CU ヘッダの構造、行番号ヘッダのファイル名表現、ロケーションリストの形式などが版ごとに違うため、実務のツールは `version` を見て読み方を切り替えます [](#cite:eager2012)。
- **32 ビット形式と 64 ビット形式**: これは対象アーキの 32/64 とは別の話で、DWARF 内部のオフセット幅の区別です。初期長フィールドが `0xffffffff` で始まると 64 ビット形式（オフセットが 8 バイト）に切り替わります。巨大なデバッグ情報で必要になります。

## ベンダ拡張と言語ごとの表現

DWARF は拡張可能な形式で、コンパイラ独自のタグ・属性（`DW_TAG_GNU_*`, `DW_AT_GNU_*` など）が多数存在します。たとえば最適化下のバックトレースを改善する**コールサイト情報** (`DW_TAG_call_site` とその引数記述) は、比較的新しく、本書では扱いませんでした。

また DWARF は C/C++ だけでなく多言語に対応しており、言語ごとに固有の表現があります。

- **C++**: 名前空間 (`DW_TAG_namespace`)、テンプレートの実体化とそのパラメータ、継承 (`DW_TAG_inheritance`)、仮想関数と vtable の記述など。
- **その他の言語**: Fortran の可変長配列、Ada の派生型、Rust の列挙体やトレイトなど、それぞれの型システムを写し取る属性群。

本書が紹介した DIE のタグ・属性・フォームは、仕様が定める全体のごく一部です。完全な一覧は仕様書 [](#cite:dwarf2017) の付録にあり、`DW_TAG_*` / `DW_AT_*` / `DW_FORM_*` / `DW_OP_*` の全コードが表として載っています。自前で DWARF を読み書きするツールを書くなら、この付録が手元の辞書になります。

---

これで、DWARF についても「本書が歩いた道」と「歩かなかった道」の全体が見渡せました。索引セクション、CFI の内部、Split DWARF、言語ごとの表現 ―― いずれも、本格的なデバッガやプロファイラを作り込んでいけば必ず出会う領域です。そのときは、本書で身につけた読み方 ―― CU ヘッダから入り、略号表を引き、状態機械を回す ―― が、そのまま土台として効きます。ここで覚えた**語**を手がかりに、仕様書の該当章へ進んでください。

理屈は十分に積み上がりました。次の第 III 部では、ここまでの知識を総動員して、ELF と DWARF を実際に読み書きする小さなツールを C 言語で作ります。手を動かして、知識を「読める・書ける」技能に変えましょう。
