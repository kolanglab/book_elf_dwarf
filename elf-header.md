# ELF ヘッダ ―― 先頭 64 バイトを読む

すべての ELF ファイルは、**ELF ヘッダ**から始まります。64 ビット ELF では、このヘッダはちょうど 64 バイトです。ここには、ファイルを解釈するために必要な最低限の情報と、2 つのテーブル（セクションヘッダ・プログラムヘッダ）への入口が書かれています。この章では、その 64 バイトをフィールドごとに、バイト位置と意味を対応させながら読み解きます。ここを読めるようになれば、ELF ファイルのどこに何があるかを自力でたどれるようになります。

## 構造体 Elf64_Ehdr の全体

ELF 仕様は、ヘッダを C の構造体として定義しています [TIS, 1995](#cite:tis1995)。64 ビット版は `Elf64_Ehdr` です。まず全体を示します。

```c
typedef struct {
    unsigned char e_ident[16];  /* マジックと識別情報 */
    uint16_t e_type;            /* ファイルの種類 */
    uint16_t e_machine;         /* アーキテクチャ */
    uint32_t e_version;         /* ELF バージョン */
    uint64_t e_entry;           /* エントリポイントの仮想アドレス */
    uint64_t e_phoff;           /* プログラムヘッダテーブルのオフセット */
    uint64_t e_shoff;           /* セクションヘッダテーブルのオフセット */
    uint32_t e_flags;           /* プロセッサ固有フラグ */
    uint16_t e_ehsize;          /* この ELF ヘッダ自身のサイズ */
    uint16_t e_phentsize;       /* プログラムヘッダ 1 エントリのサイズ */
    uint16_t e_phnum;           /* プログラムヘッダのエントリ数 */
    uint16_t e_shentsize;       /* セクションヘッダ 1 エントリのサイズ */
    uint16_t e_shnum;           /* セクションヘッダのエントリ数 */
    uint16_t e_shstrndx;        /* セクション名文字列表のインデックス */
} Elf64_Ehdr;
```

`uint16_t` は符号なし 16 ビット（2 バイト）整数、`uint64_t` は 8 バイト整数です。本書の前提（x86-64、リトルエンディアン）では、これら多バイト整数は**下位バイトが先**に格納されます。たとえば `e_type` の値 `0x0002` は、ファイル上では `02 00` というバイト並びになります。

> [!NOTE]
> **エンディアン** (endianness) とは、多バイトの整数をメモリやファイルに並べる順序のことです。下位バイトを先に置くのが**リトルエンディアン**（x86-64、ARM の多くがこれ）、上位バイトを先に置くのが**ビッグエンディアン**です。ELF はどちらにも対応でき、どちらかは後述の `e_ident[5]` に書かれています。読み間違えるとすべての数値が化けるので、解析プログラムを書くときは必ず確認します。

## e_ident ―― 先頭 16 バイトの識別情報

`e_ident` はヘッダの中でも特別で、エンディアンやクラス（32/64 ビット）に依存せず、固定の位置に固定の意味を持ちます。これは「鶏と卵」を避けるためです。エンディアンが分からなければ整数が読めませんが、エンディアンを記したフィールド自体が整数だと読めません。そこで `e_ident` は 1 バイト単位の情報だけで構成し、まずここを読んでからファイル全体の読み方を決めるのです。

| オフセット | 名前 | 意味 |
|---|---|---|
| 0 | `EI_MAG0` | `0x7F` |
| 1–3 | `EI_MAG1`–`EI_MAG3` | `'E' 'L' 'F'` (`0x45 0x4C 0x46`) |
| 4 | `EI_CLASS` | クラス: 1=32 ビット, 2=64 ビット |
| 5 | `EI_DATA` | エンディアン: 1=リトル, 2=ビッグ |
| 6 | `EI_VERSION` | ELF バージョン（常に 1） |
| 7 | `EI_OSABI` | OS/ABI 識別（0=System V, 3=Linux） |
| 8 | `EI_ABIVERSION` | ABI バージョン |
| 9–15 | `EI_PAD` | パディング（0 詰め） |

先頭 4 バイト `7F 45 4C 46`（`\x7FELF`）が[マジックナンバー](#index:マジックナンバー)です。`file` コマンドがファイル種別を判定するときも、まずこの 4 バイトを見ています。

実際に手元の実行ファイルの先頭を見てみましょう。`xxd` は16 進ダンプを表示するコマンドです。

```
$ xxd -g1 /bin/ls | head -n 1
00000000: 7f 45 4c 46 02 01 01 00 00 00 00 00 00 00 00 00  .ELF............
```

左端がファイル内オフセット、中央が 1 バイトずつの 16 進、右端が ASCII 表示です。読み取れることは次のとおりです。`7f 45 4c 46` でマジック、続く `02` で 64 ビット (`EI_CLASS=2`)、`01` でリトルエンディアン (`EI_DATA=1`)、`01` で ELF バージョン 1、残りは 0。たった 16 バイトで、これは「64 ビット・リトルエンディアンの ELF だ」と確定できました。

## ファイルの種類とアーキテクチャ

`e_ident` の次、オフセット 16 から `e_type`（2 バイト）が始まります。前章で触れたファイルの種類が、ここに数値で入ります。

| 値 | 名前 | 意味 |
|---|---|---|
| 1 | `ET_REL` | 再配置可能ファイル（`.o`） |
| 2 | `ET_EXEC` | 実行可能ファイル（固定アドレス型） |
| 3 | `ET_DYN` | 共有オブジェクト（`.so`、および PIE 実行ファイル） |
| 4 | `ET_CORE` | コアダンプ |

> [!TIP]
> 最近の Linux で普通の実行ファイルをビルドすると、`e_type` が `ET_EXEC`(2) ではなく `ET_DYN`(3) になっていることが多く、戸惑うかもしれません。これは **PIE** (Position Independent Executable、位置独立実行ファイル) がデフォルトになったためです。PIE は共有ライブラリと同じく「どのアドレスにも置ける」コードで作られており、実行のたびに配置アドレスを変える [ASLR](#index:ASLR)（アドレス空間配置のランダム化）というセキュリティ機構を効かせられます。形式上は共有オブジェクトと同じ `ET_DYN` なので、種類だけでは実行ファイルかライブラリか区別がつかないことがあります。

続く `e_machine`（オフセット 18、2 バイト）はターゲット CPU です。代表的な値は次のとおりで、x86-64 は 62 (`EM_X86_64`)、32 ビット x86 は 3 (`EM_386`)、AArch64（64 ビット ARM）は 183 です。`e_version`（オフセット 20、4 バイト）は常に 1 です。

## エントリポイントと 2 つのオフセット

ここからが ELF ヘッダの実用上の核心です。オフセット 24 から、8 バイトのアドレス／オフセットが 3 つ並びます。

- `e_entry`（オフセット 24、8 バイト）: **エントリポイント**。プログラム実行開始時に CPU が最初に実行する命令の**仮想アドレス**。`.o` や、エントリを持たないファイルでは 0。
- `e_phoff`（オフセット 32、8 バイト）: **プログラムヘッダテーブル**がファイルの先頭から何バイト目にあるか。無ければ 0。
- `e_shoff`（オフセット 40、8 バイト）: **セクションヘッダテーブル**がファイルの先頭から何バイト目にあるか。無ければ 0。

> [!NOTE]
> ここで「ファイル内**オフセット**」と「**仮想アドレス**」という 2 種類の位置が出てきました。混同しやすいので区別しておきます。**オフセット**は、ELF ファイルというバイト列の先頭からの距離（何バイト目か）です。**仮想アドレス**は、プログラムがメモリに読み込まれて実行されるときの、そのプロセスのアドレス空間上の番地です。`e_phoff` や `e_shoff` はファイルを解析するための位置なのでオフセット、`e_entry` は実行時の番地なのでアドレス、という使い分けです。両者の対応関係は、第 6 章「ローディング」で詳しく扱います。

`e_phoff` と `e_shoff` こそが、ELF ファイルを「歩く」ための出発点です。先頭 64 バイトを読めば、ここから 2 つの目次の場所が分かり、目次をたどればファイル中のあらゆるデータに到達できます。ELF が「先頭から順に読む」のではなく「ヘッダのオフセットを頼りにジャンプして読む」形式である、という点を押さえてください。

## サイズと個数を表す末尾フィールド

残りのフィールドは、2 つのテーブルを正しく読むためのサイズと個数の情報です。すべて 2 バイトです。

| オフセット | 名前 | 意味 |
|---|---|---|
| 48 | `e_flags` | プロセッサ固有フラグ（x86-64 では 0） |
| 52 | `e_ehsize` | ELF ヘッダ自身のサイズ。64 ビットでは 64 |
| 54 | `e_phentsize` | プログラムヘッダ 1 エントリのバイト数（64 ビットでは 56） |
| 56 | `e_phnum` | プログラムヘッダのエントリ数 |
| 58 | `e_shentsize` | セクションヘッダ 1 エントリのバイト数（64 ビットでは 64） |
| 60 | `e_shnum` | セクションヘッダのエントリ数 |
| 62 | `e_shstrndx` | セクション名を格納する文字列表セクションの番号 |

テーブルを配列として読むには、「先頭オフセット」「1 要素のサイズ」「要素数」の 3 つが要ります。それがちょうど `e_phoff`/`e_phentsize`/`e_phnum` と `e_shoff`/`e_shentsize`/`e_shnum` の組として与えられているわけです。たとえば i 番目のセクションヘッダは、ファイルオフセット `e_shoff + i * e_shentsize` から `e_shentsize` バイト、と計算できます。

`e_shstrndx` は少し特殊で、「セクションの**名前**（`.text` などの文字列）を格納している特別なセクションが、何番目か」を示します。セクションヘッダ自体には名前そのものは入っておらず、名前は別の文字列表に集められているためです。この仕組みは次章で詳しく説明します。

> [!CAUTION]
> セクション数やプログラムヘッダ数が非常に多い場合、2 バイトの `e_shnum` や `e_phnum` では表現しきれません。そのために ELF には「エスケープ」の規約があり、`e_shnum` が 0 のときは実際の数を 0 番セクションの `sh_size` フィールドに入れる、といった拡張が定められています。通常のプログラムでは遭遇しませんが、自前の解析ツールを堅牢にしたいときは仕様 [TIS, 1995](#cite:tis1995) の該当箇所を確認してください。

## readelf で答え合わせをする

ここまでの内容は、`readelf -h`（`-h` はファイルヘッダの意味）で人間が読める形で確認できます。

```
$ readelf -h /bin/ls
ELF Header:
  Magic:   7f 45 4c 46 02 01 01 00 00 00 00 00 00 00 00 00
  Class:                             ELF64
  Data:                              2's complement, little endian
  OS/ABI:                            UNIX - System V
  Type:                              DYN (Position-Independent Executable file)
  Machine:                           Advanced Micro Devices X86-64
  Entry point address:               0x6750
  Start of program headers:          64 (bytes into file)
  Start of section headers:          136232 (bytes into file)
  Size of this header:               64 (bytes)
  Number of program headers:         13
  Size of section headers:           64 (bytes)
  Number of section headers:         31
```

この出力の各行が、これまで見てきたフィールドにそのまま対応していることを確認してください。`Class: ELF64` は `e_ident[EI_CLASS]=2`、`Type: DYN` は `e_type=3`、`Entry point address` は `e_entry`、`Start of section headers: 136232` は `e_shoff`、`Number of section headers: 31` は `e_shnum` です。

つまり `readelf -h` がやっているのは、本章で説明したとおり先頭 64 バイトを構造体として解釈し、フィールド名を添えて表示しているだけです。第 III 部のハンズオンでは、これと同じことを行う数十行の C プログラムを実際に書きます。

ELF ヘッダを読めるようになったので、次はそこから `e_shoff` をたどった先 ―― セクションヘッダテーブルと、ファイルの大半を占める各セクションの中身へと進みます。
