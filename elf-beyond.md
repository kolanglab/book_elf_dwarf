# 扱わなかった ELF 仕様の地図

ここまでで、ELF ヘッダ、セクション、シンボルと再配置、ローディングという、ELF の「背骨」を一通りたどりました。これだけで `readelf` の主要な出力は読み解けますし、ハンズオン（第 III 部）で小さな ELF を生成・解析するのにも十分です。

しかし「はじめに」で断ったとおり、本書は ELF 仕様を網羅していません。実用上きわめて重要なのに、紙幅と一貫性のために本筋から外した領域がいくつもあります。この章は、それらを一望する地図です。各項目について、何のための仕組みか、実務でどこで出会うか、さらに調べるときの足がかりを簡潔に示します。深入りはしません。「この先に何があり、必要になったらどの語で検索し、仕様書のどこを開けばよいか」が分かれば十分です。

> [!NOTE]
> ここに挙げる仕様の多くは、純粋な ELF 規格 [](#cite:tis1995) そのものではなく、その上に積み上げられた **ABI** や **GNU/Linux 固有の拡張**に属します。ELF は「器」を定め、実際の中身（再配置タイプの意味、セクション名の慣習、ノートの形式など）は System V gABI、各アーキテクチャの psABI [](#cite:matz2013)、そして GNU ツールチェインの取り決めが埋めています。「ELF 仕様を読む」とは、しばしばこれら複数の文書を行き来することです。

## 動的リンクの周辺の仕組み

第 6 章では `.dynamic`、`PLT`、`GOT` の骨格を見ました。実際の動的リンカ (`ld.so`) は、その周辺にもっと多くの仕組みを使っています。

| 仕組み | 関連セクション／タグ | 役割 |
|---|---|---|
| シンボル検索の高速化 | `.hash`, `.gnu.hash` | 名前から動的シンボルを O(1) 近くで引くハッシュ表。大きな `libc` でも高速に解決できる |
| シンボルバージョニング | `.gnu.version`, `.gnu.version_d`(VERDEF), `.gnu.version_r`(VERNEED) | 同じ `memcpy` でも `GLIBC_2.2.5` 版と `GLIBC_2.14` 版を区別する。`symbol@@GLIBC_2.x` の正体 |
| 探索パス | `DT_RPATH`, `DT_RUNPATH` | 共有ライブラリを探すディレクトリ。`$ORIGIN` 展開など |
| 識別名 | `DT_SONAME` | ライブラリの正式名（`libc.so.6` の `.6`）。ABI 互換性の境界を表す |
| 挙動フラグ | `DT_FLAGS`, `DT_FLAGS_1` | `BIND_NOW`(遅延束縛をやめ起動時に全解決)、`PIE`、`NODELETE` など |

なかでも**シンボルバージョニング**は、実務でつまずきやすい一方、解説の少ない領域です。一つの `libc.so` の中に、API を壊さずに新旧両方の実装を同居させ、古いバイナリは古い版に、新しいバイナリは新しい版に結びつけます。この仕掛けが、Linux ディストリビューションの長期的な ABI 互換性を支えています。`readelf -V` で各バージョン定義・要求を一覧できます。動的リンクの設計思想を実装者の視点で追うなら、Drepper の文書 [](#cite:drepper2011) が決定版です。

## プログラムの始まりと終わり

`e_entry`（エントリポイント）から先、`main` に至るまで、そして `main` が終わったあとにも、いくつもの初期化・終了処理が走ります。これらを ELF はセクションと配列で記述します。

| セクション | 役割 |
|---|---|
| `.init` / `.fini` | 昔ながらの単一の初期化／終了コード片 |
| `.init_array` / `.fini_array` | 起動時／終了時に順に呼ぶ関数ポインタの配列 |
| `.preinit_array` | `.init_array` よりさらに前に呼ぶ関数の配列 |

C++ のグローバルオブジェクトのコンストラクタ、GCC の `__attribute__((constructor))` を付けた関数、`atexit` 登録の延長などは、この `.init_array`/`.fini_array` を通じて駆動されます。「`main` の前に何かが動いている」「デストラクタが呼ばれる順序」を追うときの出発点がここです。

## スレッドローカルストレージ (TLS)

`__thread int x;`（C11 では `_Thread_local`）のように、**スレッドごとに別々の実体を持つ変数**を実現するのが TLS です。これは ELF、リンカ、動的リンカ、コンパイラ、libc が協調する、かなり大きな仕組みで、本書では扱いませんでした。

- セクション: `.tdata`（初期値あり）、`.tbss`（ゼロ初期化）。`.bss` の TLS 版です。
- セグメント: `PT_TLS`。各スレッド生成時に、このテンプレートが複製されます。
- アクセス方式: **TLS モデル**（general-dynamic / local-dynamic / initial-exec / local-exec）と、専用の再配置タイプ群（`R_X86_64_TPOFF*`, `R_X86_64_DTPMOD64` など）。

> [!TIP]
> 「スレッドローカル変数のアドレスを取るとなぜ実行時関数 `__tls_get_addr` が呼ばれるのか」「なぜ TLS 変数だけ再配置の種類が異なるのか」といった疑問は、すべて TLS の仕組みに行き着きます。`x86-64` の TLS は Ulrich Drepper の "ELF Handling For Thread-Local Storage" という別文書が詳しく、本書の射程を超える深さがあります。

## 再配置タイプの全体像

第 5 章では x86-64 の代表的な再配置タイプを 3 つだけ挙げました。実際の psABI [](#cite:matz2013) は数十種類を定義しています。本書で触れなかった重要なものに、次があります。

| タイプ | 用途 |
|---|---|
| `R_X86_64_RELATIVE` | PIE/共有ライブラリのロード時、ベースアドレスを加算するだけの再配置。実行ファイル中で最も数が多いことが多い |
| `R_X86_64_GLOB_DAT` | GOT エントリにシンボルアドレスを書く |
| `R_X86_64_COPY` | 共有ライブラリ内の変数を実行ファイル側へコピーする（コピー再配置） |
| `R_X86_64_IRELATIVE` | 後述の IFUNC 用。解決関数を呼んで実体を選ぶ |

また、加数を別フィールドに持つ `Elf64_Rela`（RELA）に対し、加数を書き換え先に埋め込む `Elf64_Rel`（REL、加数なし）形式もあります。x86-64 は RELA を使いますが、i386 など REL を使うアーキテクチャもあります。

**IFUNC**（`STT_GNU_IFUNC`）は、関数の実体を実行時に選ぶ仕組みです。たとえば `memcpy` は、CPU が AVX-512 をサポートするかをロード時に判定し、最速の実装へ解決されます。`R_X86_64_IRELATIVE` を見かけたら、その背後で「解決関数が走って実装を選んでいる」と読めます。

## メタデータと堅牢化 (hardening)

実行ファイルには、コードやデータ以外に、各種のメタ情報やセキュリティ機構が埋め込まれます。

| 仕組み | 場所 | 役割 |
|---|---|---|
| ビルド ID | `.note.gnu.build-id` (`NT_GNU_BUILD_ID`) | バイナリを一意に識別するハッシュ。`debuginfo` を別ファイルで突き合わせる鍵 |
| ABI タグ | `.note.ABI-tag` (`NT_GNU_ABI_TAG`) | 必要な OS／カーネルの最小バージョン |
| GNU property | `.note.gnu.property`, `PT_GNU_PROPERTY` | CET（shadow stack / IBT）などハードウェア緩和策の要求 |
| RELRO | `PT_GNU_RELRO` | 起動後に GOT などを読み取り専用へ固める（第 6 章で予告） |
| シンボル可視性 | `Elf64_Sym.st_other` の `STV_*` | `default`/`hidden`/`protected`/`internal`。`hidden` は共有ライブラリの外へ漏らさない |

`.note.*` セクションはすべて「名前・型・記述」の三つ組という共通形式を持ち、`PT_NOTE` セグメントで実行時にも参照できます。`NT_GNU_BUILD_ID` は、分離デバッグ情報（`/usr/lib/debug` 配下や debuginfod）を引き当てるための要であり、クラッシュ解析の実務で頻繁に使います。

## 構造のバリエーション

ここまでは本書の前提（64 ビット、リトルエンディアン、x86-64）に閉じて説明してきました。その前提を外すと、別の形が現れます。

- **32 ビット ELF**: `Elf32_Ehdr`、`Elf32_Shdr`、`Elf32_Sym` など、フィールド幅の異なる別の構造体群を使います。考え方は同じですが、オフセットやサイズが変わります。`e_ident` の `EI_CLASS` で 32/64 を、`EI_DATA` でエンディアンを見分けます。
- **他アーキテクチャ**: `e_machine` が `EM_AARCH64`、`EM_RISCV` などになり、再配置タイプの体系も完全に別物になります。`e_flags` にはアーキ固有のフラグ（ARM の浮動小数 ABI、RISC-V の拡張など）が入ります。
- **圧縮セクション** (`SHF_COMPRESSED`): セクション本体を zlib/zstd で圧縮し、先頭に `Elf64_Chdr` を置く形式。デバッグ情報の `.debug_*` でよく使われ、古い慣習の `.zdebug_*` という別名もあります。
- **セクショングループ** (`SHT_GROUP`): C++ のテンプレート実体化や `inline` 関数のように、複数の翻訳単位が同じ定義を出力したとき、リンカが重複を 1 つに畳む（COMDAT）ための仕組みです。

## コアダンプという別の顔

ELF は実行ファイルだけのものではありません。プロセスが異常終了したときに生成される**コアダンプ**も、`e_type` が `ET_CORE` の ELF ファイルです。

コアダンプにはセクションヘッダは通常なく、`PT_LOAD` セグメントでクラッシュ時のメモリ内容を、そして `PT_NOTE` セグメントの中に、レジスタ状態 (`NT_PRSTATUS`)、プロセス情報 (`NT_PRPSINFO`)、補助ベクタ (`NT_AUXV`) などをノート形式で格納します。

> [!TIP]
> 「コアダンプからバックトレースを復元する」処理は、ELF（メモリとレジスタの取り出し）と DWARF（CFI によるスタック巻き戻し、行番号）の両方を総動員する好例です。本書の ELF 部と DWARF 部を読み終えると、デバッガがコアダンプをどう読んでいるのか、その全体像が見えるようになります。

## ELF の外側にある関連仕様

最後に、ELF ファイルそのものには書かれないが、ELF を扱ううえで隣り合う話題に触れておきます。

- **リンカスクリプト**: セクションをどのアドレスにどう配置するかを指示する、リンカへの設定言語。組み込みやカーネルのレイアウト制御で必須です（ELF 規格ではなくリンカの機能）。
- **PIE と静的リンク**: 位置独立実行ファイル (PIE) と、すべてを 1 つに固める静的リンクの違い。`ET_DYN` と `ET_EXEC` の使い分け、起動時再配置の有無に関わります。
- **補助ベクタ (auxv)**: カーネルが実行開始時にスタックへ積む、ページサイズや乱数シードなどの情報。`PT_LOAD` の手前、ローダとカーネルの境界に位置します。

これらを深追いするには、各アーキテクチャの psABI、System V gABI、リンカ実装者の連載 [](#cite:taylor2007) や Levine の古典 [](#cite:levine2000) が次の一歩になります。

この地図で、ELF の「本書が歩いた道」と「歩かなかった道」の全体が見えたはずです。歩かなかった道の多くは、シンボルバージョニング、TLS、IFUNC、コアダンプのように、いざ必要になると避けて通れないものばかりです。そのときは、ここで覚えた**語**を手がかりに仕様書へ分け入ってください。本書で身につけた「バイト列を切り分ける」読み方が、そのまま通用します。
