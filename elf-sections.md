# ファイルを区分けするセクション

ELF ヘッダの `e_shoff` をたどると、**セクションヘッダテーブル**にたどり着きます。セクションは、リンクビューにおけるファイルの基本的な区分け単位でした。この章では、セクションヘッダの構造を 1 フィールドずつ読み解き、続いて `.text` や `.bss` といった主要セクションが何を入れる箱なのかを具体的に押さえます。セクションが分かると、`readelf -S` の出力がすべて意味を持って見えてきます。

## セクションヘッダ Elf64_Shdr

各セクションは、`Elf64_Shdr` という 64 バイトの構造体 1 つで記述されます [](#cite:tis1995)。セクションヘッダテーブルは、この構造体の配列です。

```c
typedef struct {
    uint32_t sh_name;       /* セクション名（文字列表へのオフセット） */
    uint32_t sh_type;       /* セクションの種類 */
    uint64_t sh_flags;      /* 属性フラグ */
    uint64_t sh_addr;       /* 実行時の仮想アドレス（無ければ 0） */
    uint64_t sh_offset;     /* ファイル内オフセット */
    uint64_t sh_size;       /* セクションのバイト数 */
    uint32_t sh_link;       /* 関連する別セクションの番号 */
    uint32_t sh_info;       /* 種類ごとの補助情報 */
    uint64_t sh_addralign;  /* アラインメント要求 */
    uint64_t sh_entsize;    /* 固定長エントリのサイズ（表のとき） */
} Elf64_Shdr;
```

注目すべきは `sh_name` が**文字列そのものではなく整数**である点です。これは前章で触れた「名前は別の文字列表に集める」設計の現れで、`sh_name` は名前文字列の格納場所を指すオフセットになっています。この仕組みは後の節で詳しく見ます。

`sh_offset` と `sh_size` の組で、このセクションの実体がファイルのどこに何バイトあるかが決まります。`sh_addr` は、そのセクションが実行時にメモリに置かれるならその仮想アドレス、置かれない（リンク専用の）セクションなら 0 です。

## セクションの種類 sh_type

`sh_type` は、そのセクションが何を入れる箱なのかを示します。主要なものは次のとおりです。

| 値 | 名前 | 意味 |
|---|---|---|
| 0 | `SHT_NULL` | 無効。0 番セクションは常にこれ |
| 1 | `SHT_PROGBITS` | プログラムが定めた中身（コード・データなど） |
| 2 | `SHT_SYMTAB` | シンボルテーブル |
| 3 | `SHT_STRTAB` | 文字列テーブル |
| 4 | `SHT_RELA` | 加数付き再配置情報 |
| 8 | `SHT_NOBITS` | ファイル上に実体を持たない（`.bss`） |
| 11 | `SHT_DYNSYM` | 動的リンク用シンボルテーブル |

> [!IMPORTANT]
> `SHT_NOBITS` は重要な特例です。普通のセクションはファイル上に実体を持ちますが、`SHT_NOBITS` のセクションは**ファイル上に 1 バイトも実体を持ちません**。`sh_size` は「メモリ上で占めるべきサイズ」を表すだけで、`sh_offset` の位置に実データはありません。後述する `.bss`（ゼロ初期化データ）がこれにあたります。0 で埋めるだけの領域をファイルに記録するのは無駄なので、サイズだけ記録しておき、ローダがメモリ上で 0 埋めするのです。

## セクションの属性 sh_flags

`sh_flags` は、そのセクションのメモリ上での扱いをビットフラグで表します。前章のメモリ保護属性と密接に関係します。

| ビット | 名前 | 意味 |
|---|---|---|
| 0x1 | `SHF_WRITE` | 実行時に書き込み可能 |
| 0x2 | `SHF_ALLOC` | 実行時にメモリを占有する |
| 0x4 | `SHF_EXECINSTR` | 実行可能な機械語を含む |

たとえば `.text`（コード）は `SHF_ALLOC | SHF_EXECINSTR`（メモリを占有し、実行可能）、`.data`（書き換え可能データ）は `SHF_ALLOC | SHF_WRITE`、`.rodata`（読み取り専用データ）は `SHF_ALLOC` のみ、という具合です。`SHF_ALLOC` が立っていないセクション（たとえばシンボルテーブルやデバッグ情報）は、ファイルには存在しても実行時にはメモリに読み込まれません。デバッグ情報をいくら積んでも実行時メモリを消費しないのは、このためです。

## 文字列テーブルとセクション名の解決

`sh_name` がオフセットだという話に戻りましょう。ELF では、名前文字列をあちこちに散らばせず、**文字列テーブル** (string table) という専用セクションにまとめて格納します。文字列テーブルは、ヌル文字 (`\0`) 区切りの文字列を連結しただけの単純なバイト列です。先頭は必ずヌル文字で始まります。

たとえば文字列テーブルの中身が次のようなバイト列だとします（`\0` を区切りとして表示）。

```
オフセット: 0    1         7        13
内容:       \0 . t e x t \0 . d a t a \0 ...
```

このとき、`sh_name = 1` のセクションは、オフセット 1 から始まるヌル終端文字列 `.text` という名前を持つ、と解決されます。`sh_name = 7` なら `.data` です。「オフセットから始めて、ヌル文字までを読む」というのが文字列テーブルの使い方です。

セクション名を格納する文字列テーブルは特別で、`.shstrtab`（section header string table）という名前で存在し、その**セクション番号**が ELF ヘッダの `e_shstrndx` に書かれていました。したがってセクション名を解決する手順は次のようになります。

1. ELF ヘッダの `e_shstrndx` から、`.shstrtab` が何番セクションかを知る。
2. その番号のセクションヘッダを読み、`sh_offset` から `.shstrtab` 本体の位置を得る。
3. 各セクションの `sh_name` を、その本体内のオフセットとして使い、名前文字列を取り出す。

> [!NOTE]
> 文字列テーブルは ELF の随所で使い回される基本部品です。セクション名用の `.shstrtab` のほかに、後章で扱うシンボル名用の `.strtab`、動的シンボル名用の `.dynstr` があります。どれも「ヌル終端文字列の連結」という同じ構造で、参照する側がオフセットを持つ、という同じパターンです。このパターンは ELF だけでなく DWARF でも繰り返し現れるので、ここでしっかり理解しておくと後が楽になります。

## よく使う主要セクション

実際の実行ファイルに現れる代表的なセクションを、役割ごとに整理します。これらの名前は ABI で慣習的に定められており [](#cite:tis1995)、ツールやリンカスクリプトが特定の名前を前提にしています。

| セクション | 種類 | 内容 |
|---|---|---|
| `.text` | PROGBITS | 機械語のコード本体 |
| `.rodata` | PROGBITS | 読み取り専用データ。文字列リテラルや `const` 配列など |
| `.data` | PROGBITS | 初期値が非ゼロのグローバル／静的変数 |
| `.bss` | NOBITS | 初期値がゼロのグローバル／静的変数 |
| `.symtab` | SYMTAB | シンボルテーブル（次章） |
| `.strtab` | STRTAB | `.symtab` のシンボル名 |
| `.shstrtab` | STRTAB | セクション名 |
| `.rela.text` | RELA | `.text` に対する再配置情報（次章） |
| `.debug_*` | PROGBITS | DWARF デバッグ情報（第 II 部） |

`.data` と `.bss` の区別は、初学者がよくつまずくところです。`int counter = 5;` のように非ゼロで初期化したグローバル変数は `.data` に置かれ、その初期値 `5` がファイルに記録されます。一方 `int total;` や `int buf[1000];` のようにゼロ初期化される（C では明示しなくてもグローバル変数は 0 初期化される）変数は `.bss` に置かれます。`.bss` は `SHT_NOBITS` なので、何万バイトの配列でもファイルサイズは 1 バイトも増えません。実行時にローダがまとめてゼロ埋めしてくれるからです。

> [!TIP]
> 大きな配列をグローバルに `static char buffer[10*1024*1024];` のように確保しても実行ファイルが 10 MB 太らないのは `.bss` のおかげです。逆に `static char buffer[10*1024*1024] = {1};` のように非ゼロ初期化すると `.data` 行きになり、ファイルが実際に肥大化します。この違いはバイナリのサイズを気にする場面で効いてきます。

## readelf -S で確認する

`readelf -S`（`-S` はセクションヘッダの意味）で、ここまでの内容を一覧できます。

```
$ readelf -S /bin/ls
There are 31 section headers, starting at offset 0x21428:

Section Headers:
  [Nr] Name              Type             Address           Offset
       Size              EntSize          Flags  Link  Info  Align
  [ 0]                   NULL             0000000000000000  00000000
       0000000000000000  0000000000000000           0     0     0
  [13] .text             PROGBITS         0000000000004f50  00004f50
       0000000000010f81  0000000000000000  AX       0     0     16
  [25] .bss              NOBITS           00000000000380e0  000370e0
       00000000000012a0  0000000000000000  WA       0     0     32
```

`[13] .text` の行を読むと、種類が `PROGBITS`、`Flags` が `AX`（`SHF_ALLOC | SHF_EXECINSTR`）、つまり実行可能なコードだと分かります。`[25] .bss` は種類 `NOBITS`、`Flags` が `WA`（書き込み可・占有）で、`Size` は `0x12a0` あるのに、その分のファイルは存在しません。`[0]` 番が常に空の `NULL` セクションである点も確認できます。

セクションは「リンクビュー」の単位でした。次章では、そのリンクビューの中心的な目的（名前とアドレスを結びつける**シンボル**と**再配置**）に踏み込みます。これはリンカが実際に何をしているのかを理解する鍵であり、続く第 6 章のローディングともつながっていきます。
